# CLAUDE.md — Agent Hub

## What is Agent Hub

Agent Hub is a desktop application that orchestrates the SDD (Spec-Driven Development) workflow using Claude Code CLI. It is NOT an IDE or editor — it is a **visual manager** for the lifecycle of development tasks that run on local projects.

The app:
- Sends enriched prompts to Claude Code CLI
- Streams real-time output (streaming → logs)
- Manages task state (phases, pauses, re-runs)
- Learns from code reviews to improve future tasks
- Reads/writes Claude Code configuration files (settings.json)

All code generated by Claude Code lives INSIDE each project (path). Agent Hub is only the orchestrator.

## Tech Stack

- **Framework**: Electron (cross-platform desktop: Ubuntu, macOS, Windows)
- **Frontend**: React 19 + TypeScript + TailwindCSS + Vite
- **Database**: SQLite (via better-sqlite3)
- **CLI Integration**: Claude Code CLI (subprocess)
- **Plugin Transport**: MCP (Model Context Protocol) — universal integration layer
- **Real-time logs**: Electron IPC (main → renderer)

## SDD Workflow — Dynamic Phases

The workflow has two layers: **core phases** (always present) and **plugin phases** (added by active plugins).

### Core Phases (fixed, always run)

```
Phase 0 — Spec Review
  hooks: on:before_spec → on:spec_needs_input | on:spec_complete
  The agent analyzes the spec. If incomplete → pauses with suggestions.

Phase 1 — Plan
  hooks: on:plan_ready → on:plan_approved
  Decompose into subtasks, identify risks, present plan.

Phase 2 — Implement (TDD)
  hooks: on:implement_complete
  Code + tests. Red → Green → Refactor. Incremental.

Phase 3 — Quality Gate — Automatic loop
  hooks: on:review_started → on:quality_fail | on:quality_pass → on:quality_max_loops → on:core_complete
  Tests pass → IA Review → Fix if issues → Re-check.
```

### Plugin Phases (optional, from code-hosting plugin)

```
Phase 4 — Ship (capability: "ship")
  hooks: on:ship_started → on:ship_failed | on:pr_created
  Conventional commit + push + PR/MR.

Phase 5 — PR Feedback (capability: "pr_feedback")
  hooks: on:pr_changes_requested → on:pr_fix_pushed | on:pr_approved → on:task_complete
  Waits for human review. User clicks "Fetch & Fix" or "Approve".
```

Without a code-hosting plugin, the workflow ends at Phase 3. The user handles git/PR manually.

### Workflow Hooks (25 total)

All hooks are fired via `fireHook()` from the plugin engine. Plugins subscribe to hooks in their `manifest.json`.

**Lifecycle (4):**
- `on:workflow_started` — fired when the SDD workflow begins (with startPhase)
- `on:task_complete` — fired when a task completes successfully (Phase 3 without code-hosting, or PR approved)
- `on:workflow_failed` — fired on unrecoverable error (with error message)
- `on:workflow_aborted` — fired when user stops the agent

**Phase 0 — Spec Review (3):**
- `on:before_spec` — before spec analysis (enrichment: PM plugin injects requirement data)
- `on:spec_needs_input` — spec incomplete, pausing for user input (with suggestions)
- `on:spec_complete` — spec analysis passed

**Phase 1 — Plan (2):**
- `on:plan_ready` — plan generated, awaiting user approval (with planSummary)
- `on:plan_approved` — user approved the plan (PM plugin can create dev_tasks)

**Phase 2 — Implement (1):**
- `on:implement_complete` — implementation phase finished (PM plugin can mark subtasks done)

**Phase 3 — Quality Gate (5):**
- `on:review_started` — quality review loop iteration started (with reviewLoop)
- `on:quality_fail` — review found issues (with issues detail)
- `on:quality_pass` — review passed
- `on:quality_max_loops` — max review iterations reached without pass
- `on:core_complete` — all core phases finished

**Phase 4 — Ship (3):**
- `on:ship_started` — ship phase beginning
- `on:ship_failed` — ship phase failed (with error)
- `on:pr_created` — PR/MR created (with prNumber, branchName)

**Phase 5 — PR Feedback (3):**
- `on:pr_changes_requested` — reviewer left comments (with commentCount)
- `on:pr_fix_pushed` — fixes pushed after review cycle (with reviewLoop)
- `on:pr_approved` — user approved the push (with prNumber)

### Global (read from ~/.claude/settings.json)

- ramziddin/tdd (TDD)

- callstack/github (Git & GitHub Workflow)
- alinaqi/sdd (Spec-Driven Development)

### Per-Project (read from {project.path}/.claude/settings.json)
Optional skills toggled on/off per project:
wordpress-pro, react-patterns, nestjs-patterns, flutter-skills, etc.

The app reads and writes these files directly. Toggle ON = adds to settingSources[]. Toggle OFF = removes.

## Plugin System

Agent Hub uses a capability-based plugin system. Plugins extend the workflow without modifying core code. See `docs/PLUGIN-DEVELOPMENT.md` for the complete creation guide.

### Plugin Contributions
- **Hooks**: react to workflow events (e.g., on:quality_pass → PM marks criteria met)
- **Phases**: add phases to workflow (e.g., code-hosting adds Ship + PR Feedback)
- **Enrichment**: inject data into existing phases (e.g., PM pre-fills specs)
- **Actions**: add manual buttons to UI (e.g., "Sync with PM")
- **Operations**: reusable MCP tool call templates (listMyWork, fetch, updateStatus)
- **Task Fields**: inject dynamic fields into TaskForm (e.g., PM adds a requirement selector before the title field)

### Plugin Levels
- **Level 1** (declarative): JSON manifests + MCP — for PM tools, Notifications, CI/CD, Docs
- **Level 2** (adapter): TypeScript module — for Code Hosting (GitHub, GitLab, Bitbucket)

### Plugin Files
- `plugin.json`: identity, requirements, config schema (generates UI form), task fields (dynamic form injection)
- `manifest.json`: workflow hooks, operations, field mappings, status mappings
- `setup.json`: installation steps (MCP configuration, CLI checks)

### Plugin Storage
- Plugin files: `~/.config/agent-hub/plugins/{id}/`
- Plugin registry: `~/.config/agent-hub/plugins/installed.json`
- MCP servers: `~/.claude.json` (configured via plugin installer)
- Secrets: encrypted in Agent Hub SQLite

## Learning System — Knowledge Base

Every code review (AI or human) that finds and fixes an issue:
1. Gets registered as a pattern in SQLite (knowledge_entries)
2. Gets exported to a .md file per project (knowledge/{project}.md)
3. Gets automatically injected into the prompt of subsequent tasks
4. Gets tracked by how many times it has been applied (times_applied)

## Database — SQLite

Main tables:
- projects: id, name, path, repo, description, optional_skills[], test_command, code_hosting, plugin_pm, plugin_pm_config
- tasks: id, project_id, title, description, acceptance_criteria[], images[], model, status, pr_number, review_cycle, spec_suggestions[], plan_summary, branch_name, criteria_status, pm_work_item_id, pm_work_item_url
- agent_runs: id, task_id, phase, started_at, finished_at, result, output, error_output
- logs: id, task_id, project_name, message, kind, created_at
- knowledge_entries: id, project_id, category, severity, title, description, code_example, anti_pattern, times_applied
- review_patterns: id, knowledge_id, task_id, reviewer, issue_found, fix_applied, auto_fixable
- settings: key, value

## Development Rules

- Follow SDD: spec first, then implement
- TDD: tests before code
- TypeScript strict across the entire project
- Functional React components with hooks
- TailwindCSS for styling (no custom CSS)
- Electron IPC for main↔renderer communication (NEVER nodeIntegration: true)
- SQLite sync in main process, async IPC to renderer
- REFERENCE-UI.jsx contains the complete functional prototype — use it as visual and logic reference, but implement with the real architecture (separate components, TypeScript, SQLite, IPC)

## Task Editing Warnings

- Queued → free editing, no warning
- PR Feedback → warning: code and PR already exist, editing implies full re-execution
- Completed → red warning: task will be reopened, returns to queue
- Saving an already started task → status resets to "queued"

## User Prerequisites

- Claude Code CLI installed and authenticated (`claude login`)
- Git configured with user name and email
